Hive的学习笔记

Hive主要针对多读少改的数据仓库应用，但我们不能因此就草草将Hive理解成“数据库”。它不存储数据，反而更像一个解析器——它能够把类SQL语言解析成MapReduce程序，在Hadoop的框架基础上进行大规模的计算。
从结构上而言，Hive和数据库除了拥有相似的结构化查询语句之外，就没有什么相同的地方了，甚至它对数据的查询效率还要远远低于（合适规模数据下）秒级毫秒级响应的常用数据库。
主要差别如下：

  1. 基于HDFS，Hive不支持对数据的随机修改。
  2. Hive没有索引，查询数据需要遍历整张表；同时其默认计算引擎MapReduce十分耗时，因此具有很高的执行延迟。
  3. Hive处理的数据规模要比传统关系型数据库大，数据量大时更适合。【关系型数据库适合储存作用关系明确的业务数据】

为了支持多会话，我们将Hive元数据保存至MySQL数据库。MySQL默认情况下只允许本机ip用root用户连接，因此我们需要修改mysql库下user表中user栏为root的那条记录的host为”%”。

我们再用flush privileges，让user表的更改生效。这样就可以在其他ip用jdbc客户端远程连接了。

log4j的Jar包冲突问题

由于Hadoop和Hive都自带日志服务，而Hive在执行时需要读一遍Hadoop的配置；为了不引起log4j的Jar包版本冲突问题，可以删除Hive下的log4j-slf4j-impl文件。

metastore和hiveserver2服务

metastore负责维护hive的元数据，由它全权负责和mysql交互。其实在上述配置中，如果选择不配置hive.metastore.uris，那么将是hive自己来和mysql进行交互。
hiveserver2服务支持jdbc的连接，允许其他用户通过jdbc操作hive。hiveserver2在hive本地客户端运行时可以不启。
启动metastore和hiveserver2会阻塞当前窗口，并生成两个RunJar进程。而且如果我们关掉这两个被阻塞的窗口，这两个进程也会自动挂掉，因此默认情况下需要我们另起会话进行hive操作。
但，我们可以通过nohup告诉关窗口时不停止该窗口启动的进程，并采取&来取消窗口阻塞。
nohup命令会打印一些信息，我们可以使用重定向把这些信息统一放到一个文件里。

管理表(MANAGED)和外部表(EXTERNAL)

默认情况下，创建的表都是管理表。管理表在删除时，其HDFS上连带的源数据也会被删除，而删除一张外部表仅仅是删除了表结构和映射关系，并不会删除HDFS上的源数据文件。
在大量查询计算的业务场景中外部表更加常用。多个用户可以在外部表的基础上通过SELECT+INSERT把中间、结果表设置成管理表，这样既能保证数据安全不丢失，也能提高数据的运用频率。
注意：外部表和管理表是可以通过alter table student set tblproperties('EXTERNAL'='TRUE'/'FALSE');互相转换的。但需要小心，转换的时候属性名和属性内容都必须大写!
